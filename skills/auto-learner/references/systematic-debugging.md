# Systematic Debugging - 系统化调试方法论

## 核心原则

**永远先找根本原因，再尝试修复！**

症状修复 = 失败。

## 四个阶段

### Phase 1: 根本原因调查

在尝试任何修复之前：
- 仔细阅读错误信息
- 完整阅读堆栈跟踪
- 记录行号、文件路径、错误码
- 尝试稳定复现问题
- 检查最近的改动（git diff）

### Phase 2: 模式分析

修复前先找模式：
- 寻找类似工作的代码
- 对比参考实现
- 找出差异点

### Phase 3: 假设和测试

科学方法：
- 形成单一假设："我认为 X 是根本原因，因为 Y"
- 最小改动测试
- 验证后再继续

### Phase 4: 实现

修复根本原因，不是症状：
- 只修复必要的部分
- 验证修复有效
- 避免引入新问题

## 调试命令示例

```bash
# 检查环境变量
echo "=== 环境变量 ==="
env | grep KEY

# 检查系统状态
echo "=== 进程状态 ==="
ps aux | grep node

# 检查日志
echo "=== 最新错误 ==="
tail -n 50 error.log

# 网络检查
curl -v https://example.com
```

## 何时使用

- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别是在：**
- 时间紧迫时（紧急情况容易猜测）
- "就一个快速修复"看起来很明显时
- 你已经尝试过多个修复
- 之前的修复没起作用
- 你不完全理解这个问题
